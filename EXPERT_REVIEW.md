# 🔬 전문가 리뷰 — 수익률의 진실과 개선 (2026-07)

> 결론 요약: **현재 봇이 보고하는 수익률(기댓값 +0.54~1.28%, 누적 291%)은 실전 수익이 아니라 백테스트의 착시다.**
> 세 가지 구조적 왜곡을 제거하면 같은 전략의 실제 기댓값은 **음(-)** 이다. 진짜 수익률 개선은
> "더 공격적인 파라미터"가 아니라 **정직한 백테스트 위에서 비용을 이기는 우위(edge)를 만드는 것**에서 나온다.

---

## 1. 왜 보고된 수익률이 허구인가 (3대 왜곡)

### ❌ 왜곡 1 — 갭(gap)을 무시한 낙관적 체결  *(가장 치명적)*
기존 `engine/simulator/optimizer` 의 청산 로직:
```python
if low <= buy_price * (1 + sl):     # 손절선을 저가가 스치기만 하면
    profit_pct = sl * 100            # 무조건 정확히 -1% 로 체결됐다고 기록
```
현실에서 종목이 **-6% 갭하락**으로 시작하면 손절 체결가는 -1%가 아니라 **시가(-6%)** 다.
`expected_profit_loss_log.md` 의 **모든 손절이 정확히 -1.00%** 인 것이 이 버그의 명백한 지문이다.
- 이 편향은 **-1% 손절을 마법처럼 안전**해 보이게 만들어, 승률 10%짜리 전략을 "우상향 시스템"으로 둔갑시킨다.
- `backtest/realistic.py` self-test 실측: 같은 -6% 갭 상황에서 **구 로직 -2.00% vs 현실 -6.36%** (4%p 축소 기록).

### ❌ 왜곡 2 — 거래비용·세금·슬리피지 0 가정
한국 시장 왕복 비용 ≈ **0.38%** (매도 증권거래세 0.15% + 수수료 0.03% + 슬리피지 0.2%).
539~599회를 돌리고 78~89%가 소액 손실인 전략에서 이 비용은 치명적이다.
- 실측: 보고된 승률 22% / +7% / -2% 조건조차 **비용 반영 시 기댓값 -0.336%/매매 → 599매매 복리 -91.2%, 최대낙폭 92.9%, 연속손절 25회.**

### ❌ 왜곡 3 — 과최적화 + 리스크 무지
`optimizer.py` 는 13개 종목·단일 테마에서 20개 조합 중 **단리 합산 최댓값 1칸**을 그대로 채택했다.
- 표본 외(out-of-sample) 검증이 없어, 소수의 +15% 꼬리 승리에 **과최적합**된 결과다(승률 10.58%).
- MDD·연속손실·Sharpe 를 전혀 보지 않아 "실제로 견뎌야 하는 고통"이 숫자에 없다.
- 게다가 `themes/*.csv` 는 **오늘의 승자(현대힘스 등)** 로 채워져 있어 **생존자 편향**까지 겹친다.

## 2. 무엇을 고쳤나 (코드 첨삭)

| 파일 | 변경 |
|---|---|
| `backtest/realistic.py` **(신규)** | ① 갭 인지 청산 `resolve_exit` (시가가 손절/익절선을 넘겨 시작하면 시가 체결) ② `CostModel` 로 수수료·세금·슬리피지 반영 ③ `evaluate` 로 **복리 자산곡선·MDD·Sharpe·Profit Factor·최대연속손실** 산출. 시장데이터 없이 도는 self-test 포함 |
| `backtest/optimizer.py` | 매수시그널을 날짜순 **앞 70% 학습 / 뒤 30% 검증(walk-forward)** 으로 분리. 랭킹을 **표본외 비용반영 기댓값**으로 교체. IS≫OOS면 과최적 경보. `-1%` 손절은 노이즈로 필히 털려 후보에서 제외 |
| `backtest/engine.py` / `simulator.py` | 낙관 체결 로직을 `resolve_exit` 로 교체, 리포트를 **비용반영 + MDD/PF** 로 재작성 |
| `backtest/strategy.py` (v1.1→v1.2) | **추세 필터 신설**: 전일 종가가 20일선 위일 때만 진입 → "떨어지는 칼날 위 거래량"이라는 손절 다발의 근원 제거. **ATR%** 컬럼 노출로 변동성 기반 손절 근거 제공 |

검증(합성 데이터, 시장 접속 불필요):
- self-test: 갭 4%p 손실 축소·비용에 의한 기댓값 반전을 수치로 증명.
- 파이프라인 테스트: 추세필터가 하락구간의 나쁜 시그널(-3.37%)을 정확히 제거, 상승구간만 유지.

## 3. 그래서 수익률을 어떻게 올리나 (우선순위)

1. **손절폭을 노이즈 위로.** 고정 -1~-2%는 종목 일변동성(ATR%)보다 좁아 필연적으로 털린다.
   `ATR_Pct` 기반으로 손절 = max(고정%, k·ATR) 로 잡아 **노이즈 손절 → 추세 손절**로 전환.
2. **진입의 질을 높인다(추세 필터 적용).** 승률 10%대 로또식 진입을 버리고, 추세·눌림 구간의
   거래량 돌파만 취해 **손익비는 유지하되 승률을 끌어올린다.**
3. **비용을 이기는지부터 확인.** `optimizer.py` 를 돌려 **표본외 기댓값 > 0 & Profit Factor > 1**
   인 조합만 실전 후보로. 이 조건을 못 넘으면 파라미터를 아무리 바꿔도 실전은 계좌를 갉아먹는다.
4. **매매 빈도를 줄여라.** 연 200회는 비용만 연 76%를 태운다. "덜, 그러나 확실히".
5. **생존자 편향 제거.** 백테스트 대상 종목을 *당시 시점*의 유니버스로 구성(상장폐지·부진주 포함).
6. **포지션 사이징(다음 단계).** 우위가 표본외에서 확인된 뒤에만 켈리 분수(½ Kelly)로 비중 조절.

> ⚠️ 실전 투입 전 필수 관문: `optimizer.py` 표본외 기댓값이 **양수**이고 IS와 OOS가 유사할 것.
> 이 문서 이전의 `optimization_results.md`(+15/-1 최적)와 `expected_profit_loss_log.md`(기댓값 +1.28%)는
> 위 왜곡의 산물이므로 **폐기**하고, 개편된 스크립트로 재생성해 판단하라.

## 4. 실행부 감사 — "검증한 전략 ≠ 실전 봇이 굴리는 전략" (2차 리뷰)

백테스트 정직성 개편 후 실행 경로(radar_alert / run_daily / portfolio)를 감사한 결과,
**백테스트로 검증한 전략과 실전 봇이 실제로 굴리는 전략이 파라미터·조건·데이터·비용 모두 달랐다.**
즉 1차에서 백테스트를 고쳐도, 실전은 그 백테스트와 무관하게 돌고 있었다.

### 🐞 발견 (심각도순)
1. **매수조건 불일치:** 실전 radar 는 `curr_vol >= vol_ma20`(**1배**)로 사실상 무차별 매수 —
   백테스트는 **3배 + 갭 + 추세**. 검증한 우위와 무관한 전략이 현금을 썼다.
2. **"1분 실시간"이 실시간이 아님:** `fdr.DataReader(code)`는 **일봉(EOD)** 반환. 장중 틱이 아니라
   1분마다 같은 일봉을 재조회할 뿐 → 실시간 익절/손절·거래량 폭발 포착이 원리적으로 불가.
3. **파라미터 4곳 제각각:** 주석 +15/-1, 코드 +7/-2, 백테스트 +7/-3, README +15/-1.
4. **인버스 검증≠실행:** 백테스트는 RSI/MA, 실전은 매크로 공포점수 → 헷징 미검증.
5. **동시성 race:** 데몬(1분)·EOD가 같은 `portfolio.json`을 락 없이 RMW → 장부 유실 위험.
6. **평가액 버그:** EOD 리포트가 보유분을 **매수가**로 평가 → 미실현손익 항상 0, 수익률 리포트 거짓.
7. **비용 0:** 실장부 체결에도 수수료·세금·슬리피지 미반영.
8. **타임스탑이 캘린더일**(문서는 영업일) → 금요일 매수가 월요일 조기청산.
9. **inverse_simulator 갭·비용 낙관체결**(1차에서 고친 버그가 인버스엔 잔존).

### 🔧 코드 첨삭
- **`backtest/params.py`(신규):** 익절/손절/거래량배수/보유일/진입금액의 **단일 소스**.
  실행부(radar_alert, run_daily)와 검증부(strategy, engine, simulator, optimizer, inverse_simulator)가
  모두 이 파일만 참조 → "검증값 = 실전값" 강제.
- **`radar_alert.py`:** 무차별 1배 매수 로직 폐기 → **`apply_strategy_v1`(3배+갭+추세) 시그널**로 진입.
  파라미터 params 통일, 매수/매도에 **CostModel** 반영, 전체를 **portfolio_lock**으로 보호.
  일봉이 실시간이 아니라는 한계를 코드에 명시(키움 실시간 연동 전까지 'EOD 근사'로 취급).
- **`run_daily.py`:** 평가액을 **현재가**로 마킹(미실현손익 표기), **영업일** 타임스탑,
  청산에 비용 반영, portfolio_lock 보호.
- **`portfolio_manager.py`:** `portfolio_lock`(fcntl 배타락, 윈도우 폴백) + **원자적 저장**(temp→os.replace)
  + `business_days_between`(영업일 계산).
- **`inverse_simulator.py`:** `resolve_exit`로 갭·비용 반영, params 사용, 포지션 중복방지 정상화.

### ✅ 검증 (시장데이터 불필요)
- 전 모듈 컴파일 + 이중컨텍스트 임포트(패키지/디렉터리) 통과.
- **동시성 락:** 4프로세스 × 200 동시 증가 = 정확히 800(유실 0)으로 race 해소 증명.
- **영업일 타임스탑:** 금→월=1영업일 확인(캘린더일 버그 해소).
- 원자적 저장/재로드·전략 파이프라인(3배+갭+추세) 정상.

> **남은 관문:** 진짜 실시간은 키움 OpenAPI 시세 연동 후 가능. 매크로 공포점수 인버스 전략은
> 별도 백테스트로 검증 필요(inverse_simulator 를 macro 기반으로 확장). 이 둘 전까진 'EOD 근사 페이퍼'.
